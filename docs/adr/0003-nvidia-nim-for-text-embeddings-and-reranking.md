# NVIDIA NIM replaces self-hosted models for text embeddings and reranking

> **Status: not adopted (#79).** Live validation measured NVIDIA NIM's hot path (`embed_query` + rerank) at ~573ms steady-state, vs. ~281ms for the existing local `bge_m3` + `bge_reranker` setup on this machine — NIM is ~2x *slower*, not faster. The premise below ("local `mps` inference is the hot-path bottleneck") does not hold in this deployment: with models already warm in process memory and low query volume, local inference beats any remote API's ~250-350ms round-trip floor. The migration was reverted (Qdrant collection restored from a pre-migration snapshot, `bge_m3`/`bge_reranker` remain the default) and #76/#80 were closed as not planned. `nvidia_nim` remains merged and available as an opt-in provider (#77/#78) for deployments that genuinely are hardware-bound (e.g. no local GPU at all) — just not the default here. The design rationale below is kept for that use case; treat every latency-motivated claim in it as unvalidated for this deployment.
>
> This decision covers the full migration (issue #76), landing incrementally across #77 (embedding provider), #78 (reranker provider), #79 (reindex + validation), and #80 (removal of the self-hosted providers). Some consequences below only take effect once the later tickets land — each is noted.
>
> **Correction (#79):** the models named below at write time, `nvidia/llama-3.2-nv-embedqa-1b-v2` and `nvidia/llama-3.2-nv-rerankqa-1b-v2`, are wrong as of #79's live validation — the embedding model reached end-of-life on 2026-05-18 (HTTP 410) and the rerank model never resolved (HTTP 404, plus the assumed `/v1/ranking` endpoint path was itself wrong — reranking NIMs live at `https://ai.api.nvidia.com/v1/retrieval/{model}/reranking`, a different host than chat/embeddings). The corrected, live-confirmed models are `nvidia/llama-nemotron-embed-1b-v2` (2048-dim) and `nvidia/llama-nemotron-rerank-1b-v2`. The decision and rationale below are otherwise unchanged — only the specific model identifiers and the reranker invoke URL were wrong.

The hot path (`embed_query` + rerank) ran self-hosted BGE-M3 and BGE-Reranker/Qwen-Reranker on local `mps`, which was the bottleneck for query latency. We migrate text embeddings and reranking to NVIDIA NIM (`nvidia/llama-nemotron-embed-1b-v2` and `nvidia/llama-nemotron-rerank-1b-v2` — NIM's low-latency 1B generation), reusing the pattern already established for the LLM in ADR-0002. `bge_m3.py`, `nomic.py`, `qwen_embedding.py`, `sentence_transformer_base.py`, `bge_reranker.py`, and `qwen_reranker.py` are removed (#80), along with their weights. `clip_provider.py` is kept — it is the only path to image embeddings and NIM has no equivalent multimodal model in this catalog.

## Consequences

- **Full reindex required (#79).** Switching embedding model changes the vector space; the corpus is reembedded via `rebuild_embeddings.py --recreate-collection` as a one-time offline cut (no blue-green — no uptime requirement on this project). Any future embedding provider change carries the same cost.
- **Sparse vectors move to BM25.** NIM's embedqa model is dense-only, so `search_hybrid`'s sparse side falls back to BM25 — the same pattern OpenAI/Cohere/Voyage/Gemini already use in this codebase. Hybrid search stays on; dense-only search was rejected as a retrieval-quality regression nobody asked for.
- **Asymmetric failure handling on the hot path (#78).** An embedding failure fails loud (`EmbeddingError`) — there is no fallback without a dense query vector. A reranker failure degrades gracefully — the response returns chunks in raw retrieval order instead of failing the whole request, since reranking is a quality improvement, not a requirement.
- **Redis cache does NOT help the hot path (correction, #79).** `EmbeddingCacheSettings.enabled` stays `False` by default (unchanged). The reference config/`.env.example` originally documented setting `EMBEDDINGS__CACHE__ENABLED=true` for `nvidia_nim` on the theory that caching would offset repeated network latency on `embed_query` — that's wrong. `CachedEmbeddingProvider.embed_query()` deliberately never caches, by pre-existing, correct design (some providers use distinct query/document vector spaces, so caching a query vector under the document cache key would silently serve the wrong vector). Only `embed()`/`embed_passage()` — indexing-time calls, not the hot path — benefit from the cache. The reranker has no cache regardless — pairwise scores are too query/chunk-specific to reuse. See `src/core/settings.py` for the authoritative default.
- **GPU infra can be retired later (#80, follow-up).** `helm/rag-platform/values.yaml`'s `embeddingsDevice: cuda` profile and the `models` PVC exist to support self-hosted text embeddings/reranking; once this migration ships, that GPU node pool and PVC are no longer needed for text (CLIP still needs local inference for images).
